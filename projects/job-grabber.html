Job Grabber: A Command-Line Aggregator for Job Listings

Matthew Lindley Personal project, 2026

Python CLI Web Scraping Automation
Abstract A job search means checking the same handful of sites over and over for postings that match a query. Job Grabber is a Python command-line tool that searches multiple job boards at once, normalizes the results into a common schema, and exports them to a table, JSON, or CSV — so the search loop becomes one command instead of a dozen browser tabs.

Motivation

While applying to roles after graduation, I noticed the actual bottleneck wasn't finding postings — it was the repetition of opening Indeed, then LinkedIn, then re-running the same query with slightly different filters, then manually copying anything relevant into a spreadsheet. None of that requires a human in the loop. It's exactly the kind of repetitive, well-defined task that belongs in a script.

Approach

Job Grabber is structured as a small, extensible CLI built with Click. A search command takes a query, location, and source, and dispatches to one or more scraper classes that share a common interface:

  • A BaseScraper abstract class defines the contract — a search() method that returns a list of Job records — and handles shared concerns like request headers and timeouts.
  • IndeedScraper and LinkedInScraper each implement that contract by paginating through search results with requests and parsing the HTML with BeautifulSoup.
  • Every result is normalized into a single Job dataclass (title, company, location, URL, source, salary, posted date), regardless of which site it came from.
  • Results render as a formatted table in the terminal via rich, or export to .json / .csv for further filtering.

The scraper interface was the part worth getting right: adding a third source should mean writing one new class, not touching the CLI, the output formatting, or anything else.

Limitations

This is a scraper, not an API client, which means it's only as reliable as the HTML structure of the sites it targets — a markup change on Indeed's end can silently break a selector. It currently runs locally rather than on a schedule, so it's a tool I reach for deliberately rather than a background service.

Reflection

The most useful design decision was separating the scraping logic from the data model. Because every scraper returns the same Job shape, the table/JSON/CSV output code never has to know which site a result came from — and the test suite can validate the output format without making a single real HTTP request.
